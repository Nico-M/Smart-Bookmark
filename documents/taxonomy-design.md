# 书签分类体系实施方案

## 1. 背景

当前项目已经具备以下能力：

- 使用 AI 为书签生成分组路径
- 通过本地规则将一级分类收口到固定白名单
- 对 AI 返回结果进行路径归一化和兜底修正
- 对 URL 进行归一化去重

现状问题也很明显：

- 分类逻辑过度依赖当前样本，容易对“这批书签”有效、对“下一批书签”失效
- 主题、资源类型、来源形态、使用场景被混在同一棵目录树里
- [`src/shared/grouping-rules.ts`](/home/nico/Workspace/demos/bookmark_extension/src/shared/grouping-rules.ts) 中规则越来越像补丁集，长期会失控
- AI 提示词目前更像“路径约束器”，还不是“分类学驱动的语义分类器”

本方案的目标不是继续堆更多正则，而是把分类能力升级为：

`预清洗 -> 分面抽取 -> taxonomy 映射 -> 路径校验 -> 回归评估`

## 2. 设计目标

### 2.1 目标

建立一套对不同用户书签都尽量稳定的分类体系，要求：

- 一级分类稳定且数量受控
- 二三级分类来自受控词表，不依赖模型自由发挥
- 目录树只承载“主题”这一主维度
- 资源类型、来源类型、访问范围等作为辅助分面参与判断
- AI 可犯错，但最终输出必须被本地规则收口

### 2.2 非目标

本阶段不做以下事项：

- 自动批量重命名所有书签标题
- 自动删除疑似死链
- 语义级重复内容合并
- 一次性推翻当前全部分类逻辑

## 3. 核心原则

### 3.1 一棵树只承载一个主逻辑

目录树默认只表达“主题”。  
不能在同一层同时出现下列混合分类：

- `JavaScript`
- `团队博客`
- `官方文档`
- `学习教程`

上面四者分别属于不同维度：

- `JavaScript` 是主题
- `团队博客` 是来源/站点形态
- `官方文档` 是资源类型 + 来源类型
- `学习教程` 是用途 + 资源类型

如果把它们混放，同层分类天然不稳定。

### 3.2 先分面，后映射

分类时先抽取统一的语义分面，再将分面映射到受控 taxonomy。

### 3.3 受控词表优先

分类名不是临场命名题。  
一级、二级、三级分类都应优先从白名单中选取，模型只能在白名单和同义词映射范围内活动。

### 3.4 Prompt 负责理解，代码负责收口

AI 负责抽象语义，本地代码负责：

- 白名单约束
- 同义词归并
- 路径深度限制
- 禁用名称拦截
- 兜底修正
- 一致性校验

### 3.5 先开放发现，再封闭应用

taxonomy 的形成方式应是：

1. 先用样本做开放分析，发现常见主题和边界问题
2. 再固化为受控词表
3. 最后让 AI 按固定 taxonomy 应用

换句话说，探索阶段可以开放，生产阶段必须收敛。

## 4. 分面模型

建议将每条书签先抽取为以下 5 个核心分面。

### 4.1 `primary_topic`

表示“书签主要是关于什么的”，这是目录树主维度。

示例：

- 前端开发
- 后端开发
- AI 编程
- 设计素材
- 在线影视

### 4.2 `resource_type`

表示“这是什么类型的资源”。

候选值建议：

- 文档
- 教程
- 博客
- 社区
- 工具
- 视频
- API 服务
- 资讯
- 项目平台

### 4.3 `source_type`

表示“内容来源属于哪类主体”。

候选值建议：

- 官方
- 社区
- 个人
- 企业
- 公司内部
- 本地

### 4.4 `usage_intent`

表示“用户保留这条书签主要是为了什么”。

候选值建议：

- 查阅
- 学习
- 工作
- 创作
- 娱乐
- 购物

### 4.5 `scope`

表示“访问范围或环境属性”。

候选值建议：

- 公开
- 内网
- 本地
- 测试环境

## 5. 目录树与分面的关系

### 5.1 允许进入目录树的维度

默认只有以下维度可直接决定目录树：

- `primary_topic`

### 5.2 只允许辅助判断的维度

下列维度默认不直接作为主目录：

- `resource_type`
- `source_type`
- `usage_intent`
- `scope`

这些维度的作用是：

- 协助区分 `技术` 和 `学习`
- 协助决定 `AI / 编程与平台 / API 服务`
- 决定某些链接应被排除或单独归档

### 5.3 允许的例外

当主主题相同但需要稳定细分时，可在三级目录引入少量“资源子型”：

- `AI / 编程与平台 / API 服务`
- `技术 / 开发工具 / 版本控制`
- `技术 / 前端开发 / 性能优化`

这里的前提是：三级名称仍然属于受控词表，而不是自由生成。

## 6. 顶层 taxonomy 设计

### 6.1 一级分类

延续当前项目的一级白名单，保持稳定：

- 技术
- AI
- 工具
- 设计
- 学习
- 资讯
- 工作
- 生活
- 影音
- 其他主题

说明：

- `其他主题` 只作为无法判断时的兜底，不参与主动命名
- 一级分类数量不再扩张，后续变化主要发生在二三级

### 6.2 一级分类边界定义

#### 技术

放置：

- 日常开发参考资料
- 官方文档
- 技术博客
- 技术社区
- 编程语言、框架、工程化、前后端、调试工具

不放：

- 明确以系统学习为目的的课程、电子书、训练营
- 纯 AI 平台、模型、提示词资源

#### AI

放置：

- LLM、Agent、MCP、模型平台
- AI API 服务
- AI 编程平台
- AI 创作、提示词、文生图等

不放：

- 普通编程教程，仅仅因为标题里出现 Python、代码、工具等不应归入 AI

#### 工具

放置：

- 通用在线工具
- 效率工具
- 资源搜索/下载
- 软件资源
- 系统与网络工具

不放：

- 明确服务于某技术主题的开发参考内容，优先归 `技术`

#### 学习

放置：

- 系统课程
- 教材、电子书、训练营
- 成体系教程

不放：

- 零散博客、官方参考文档、社区帖子

#### 工作

放置：

- 公司内部系统
- 项目平台
- OA、Jira、Jenkins、Yapi、Confluence 等

这类内容通常还带有 `scope = 内网/测试环境` 特征。

## 8. 预清洗规则

预清洗不是删除数据，而是控制哪些书签不应直接进入“公共主题分类”。

### 8.1 应优先标记为排除或单独归档的链接

- `localhost`
- `127.0.0.1`
- `file://`
- 局域网 IP，如 `192.168.x.x`、`10.x.x.x`、`172.16.x.x ~ 172.31.x.x`
- 明显测试环境域名，如 `test.`、`dev.`、`staging.`

### 8.2 预清洗后的处理策略

建议增加字段：

- `excluded: boolean`
- `excludeReason?: string`

处理方式：

- 默认不进入 AI 公共分类主流程
- 可单独归入 `工作 / 内部系统` 或“待人工确认”
- 在预览里单独统计，不静默丢弃

## 9. Prompt 设计改造

## 9.1 现状问题

当前 [`src/background/services/ai-service.ts`](/home/nico/Workspace/demos/bookmark_extension/src/background/services/ai-service.ts) 中的提示词主要强调：

- 一级分类白名单
- 禁止某些错误一级名称
- 输出结构约束

这些约束有价值，但还缺少“科学的中间层”。

### 9.2 新 Prompt 目标

新版 Prompt 应让模型按以下顺序工作：

1. 识别书签的核心分面
2. 判断是否属于排除/归档候选
3. 根据受控 taxonomy 选择稳定路径
4. 不生成 taxonomy 之外的任意新名称

### 9.3 新 Prompt 结构建议

建议在提示词中明确：

- 目录树默认只按主题组织
- `resource_type/source_type/scope` 只作辅助判断
- 学习型资源与查阅型资源的区别
- 内网、本地、测试环境的处理方式
- 语义相近名称必须收敛

建议输出结构：

```json
{
  "groups": [
    {
      "groupPath": ["技术", "前端开发", "JavaScript"],
      "bookmarkIds": ["id-1", "id-2"]
    }
  ],
  "facets": [
    {
      "id": "id-1",
      "primary_topic": "前端开发",
      "resource_type": "文档",
      "source_type": "官方",
      "usage_intent": "查阅",
      "scope": "公开"
    }
  ]
}
```

### 9.4 Prompt 关键规则草案

建议写入提示词的关键原则：

1. 先抽取分面，再映射路径
2. 目录树只按主题分类
3. 不要把官方/社区/博客/教程与技术主题混放到同一层
4. 标题和 URL 都不足时，再参考原始路径
5. `其他主题` 只作最后兜底
6. 禁止自由发明同义词
7. 内网、本地、测试环境优先标记而不是强行混入公共主题

## 11. 文件级实施计划

### 阶段 1：文档与 taxonomy 抽离

目标：

- 先把分类知识从代码里提出来

动作：

- 新增 `src/shared/taxonomy.ts`
- 将一级/二级/三级白名单、别名映射、禁用名集中管理
- 保留现有 `grouping-rules.ts` 的外部接口，避免一次性改太大

验收：

- taxonomy 能独立阅读和维护
- 当前测试仍可运行

### 阶段 2：预清洗接入

目标：

- 不让垃圾上下文污染 AI 分类

动作：

- 在分类前增加预清洗模块
- 标记 `localhost`、内网、本地、测试环境链接
- 给预览增加排除统计

验收：

- 这类链接不会和公开内容混在同一轮 AI 分类中

### 阶段 3：Prompt 升级为“分面抽取 + taxonomy 映射”

目标：

- 让 AI 不再直接按目录名猜分类

动作：

- 改造 `buildPromptPayload`
- 增加 facets 说明
- 改造 schema，使其接受调试阶段的 facets 输出

验收：

- AI 输出分类名更收敛
- `其他主题` 占比下降

### 阶段 4：本地映射器重构

目标：

- 将当前规则文件改造成映射器而不是补丁堆

动作：

- 将顶层和二级规则迁移到 taxonomy 定义
- 将旧正则拆成：
  - alias 归并规则
  - fallback 推断规则
  - 脏名称拦截规则

验收：

- `grouping-rules.ts` 逻辑明显收敛
- 新增 taxonomy 时不必大量修改主流程

### 阶段 5：回归评估与样本集

目标：

- 防止只对当前样本有效

动作：

- 增加覆盖不同用户场景的样本数据
- 增加分面与路径的断言测试
- 对内网、本地、AI 误归类、学习/技术边界做专项测试

验收：

- 每次改动都可通过回归结果判断是否变好

## 12. 测试策略

建议新增测试类型：

### 12.1 taxonomy 映射测试

验证分面到路径的映射是否稳定。

### 12.2 预清洗测试

验证以下输入会被正确标记：

- `localhost`
- `file://`
- 内网 IP
- `test.xxx.com`

### 12.3 边界测试

重点验证：

- `技术` 与 `学习` 的边界
- `AI` 与普通编程内容的边界
- `工具` 与 `技术 / 开发工具` 的边界

### 12.4 漂移测试

验证 AI 输出的旧分类名或同义词是否能被收口到稳定 taxonomy。

## 13. 风险与控制

### 13.1 taxonomy 设计过大

风险：

- 分类名太多，维护成本爆炸

控制：

- 一级固定
- 二级少量稳定
- 三级只补高频主题

### 13.2 Prompt 过于复杂

风险：

- 模型输出不稳定，甚至偏题

控制：

- 分面字段先限制在 5 个
- 输出结构尽量简单
- 复杂解释只在调试模式使用

### 13.3 迁移期间新旧逻辑冲突

风险：

- 同一条书签在新旧体系中结果不一致

控制：

- 保留旧逻辑作为 fallback
- 先做灰度对比，不一次性切换

## 14. 当前建议执行顺序

建议按以下顺序推进：

1. 先抽离 taxonomy 配置
2. 再加入预清洗
3. 然后升级 Prompt 为分面模型
4. 再重构本地映射逻辑
5. 最后补齐回归测试与评估样本

## 15. 成功标准

若方案实施成功，应出现以下结果：

- 换一批用户书签后，一级分类仍然稳定
- `其他主题` 明显减少
- 同义分类名显著收敛
- `技术`、`学习`、`AI`、`工具` 的边界更清晰
- 本地规则文件不再持续膨胀
- 新增分类能力时主要改 taxonomy 和 prompt，而不是反复补正则

## 16. 下一步建议

下一步建议直接产出两项内容：

1. `src/shared/taxonomy.ts` 的结构设计草案
2. `src/background/services/ai-service.ts` 的新版 Prompt 草案

其中，优先级更高的是 `taxonomy.ts`，因为没有受控词表，Prompt 再漂亮也会继续漂。
